# SyncVerse Risk Assessment Microservice

AI-powered risk assessment and recommendation microservice for **AI-SyncVerse**. Analyzes project data pulled from the existing backend (never accesses the database directly), computes deterministic risk metrics, and uses a free LLM (Gemini → Groq → Hugging Face fallback chain) strictly for qualitative estimation, narrative, and recommendations — never for numeric scoring.

## Architecture

```
Backend API (existing system)
        │  REST calls only — no direct DB access
        ▼
Data Collector          (Mode 1: unified /risk-context, Mode 2: per-endpoint fallback)
        ▼
Metrics Engine          (deterministic Python calculations)
        ▼
Risk Engine             (rule-based scoring — NO LLM involvement)
        ▼
AI Context Builder      (summarizes context for the LLM)
        ▼
LLM (ai_estimators)     (AI Estimated Metrics + narrative + recommendations only)
        ▼
Report Builder          (assembles the final JSON — the only place that does)
        ▼
Internal Persistence    (SQLite by default, swappable via interface)
```

Full design rationale, database mapping, and diagrams are in `/docs` (see the architecture discussion history for the complete Phase 1/2 analysis).

## Demo Mode (test without a real backend)

Set `DEMO_MODE=true` in `.env` (this is the default in `.env.example`) to run
the full pipeline — Data Collector → Metrics → Risk Engine → LLM → Report —
against realistic built-in sample data, without needing the backend team's
API to be reachable yet:

```bash
curl -X POST http://localhost:8000/projects/demo-1/analyze
```

Set `DEMO_MODE=false` once real backend integration is ready; the service
will then use Mode 1 (`/risk-context`) with automatic fallback to Mode 2.

**Note for running tests**: make sure no `.env` file with `DEMO_MODE=true` is
present in the working directory when running `pytest` — the integration
tests exercise Mode 1/Mode 2 against a mocked backend and will be bypassed
if Demo Mode is active. `tests/` never reads `.env` intentionally, but a
`.env` in the same directory as `pytest` is picked up by `pydantic-settings`
regardless of that, so remove or rename it before testing.

## Running locally

```bash
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

cp .env.example .env
# DEMO_MODE=true works out of the box; set to false and fill in
# BACKEND_BASE_URL + at least one LLM API key for real integration

uvicorn app.main:app --reload --port 8000
```

Docs available at `http://localhost:8000/docs` (Swagger UI, auto-generated by FastAPI).

## Running with Docker

```bash
docker compose up --build
```

Service listens on port `7860` (Hugging Face Spaces' expected port).

## Running tests

```bash
pytest tests/ -v
```

All tests run offline — the backend and LLM providers are both mocked (`tests/mocks/`).

## API

| Method | Endpoint | Description |
|---|---|---|
| POST | `/projects/{id}/analyze` | Runs a full analysis and returns a new risk report |
| GET  | `/projects/{id}/risk-report` | Returns the latest risk report (regenerates it) |
| GET  | `/projects/{id}/history` | Returns previously generated reports for the project |
| GET  | `/health` | Liveness check |

## Backend integration contract

This service never touches the database. It expects the backend to expose either:

**Preferred (Mode 1):** `GET /projects/{id}/risk-context` returning project, tasks, timeline, milestones, risks, time_logs, team_members, and meetings in one payload.

**Fallback (Mode 2, used automatically if the above 404s):**
```
GET /projects/{id}
GET /projects/{id}/tasks
GET /projects/{id}/timeline
GET /projects/{id}/milestones
GET /projects/{id}/risks
GET /projects/{id}/timelogs
GET /projects/{id}/team-members
GET /projects/{id}/team-meetings
```

No database schema changes are required on the backend side.

## Design principles enforced in code

- **The LLM never outputs a numeric risk score.** `app/risk_engine/rules.py` and `aggregator.py` are the only places scores are computed, using plain arithmetic. `app/ai_estimators/estimated_metrics.py` is the only module permitted to call an LLM provider, and it only returns qualitative metrics, narrative text, and recommendations.
- **Every metric and risk category carries `source` (`calculated`/`ai_estimated`) and `confidence`.** No unlabeled numbers.
- **Graceful degradation everywhere**: a missing backend endpoint falls back to Mode 2; a failed LLM provider falls back to the next one in `LLM_PROVIDER_ORDER`; if all LLM providers fail, the report still returns with clearly low-confidence placeholders instead of a 500.
- **Stateless per-request**: no in-memory session state. The SQLite persistence layer is internal bookkeeping (history + trend snapshots), not request state.
